# コワーキングスペース運営奮闘記 - note記事作成スキル

コワーキングスペース「Tatami Works」の運営にまつわるエピソードを、ストーリー形式のnote記事として作成するスキルです。

## いつこのスキルを使うか

以下のような依頼があった場合に自動的に発動してください：

- 「奮闘記を書いて」
- 「奮闘記の記事を作成」
- 「Tatami の奮闘記」
- 「運営エピソードを書いて」
- 「Tatami Works の体験談を書いて」

## 記事の保存先

```
/home/kentaro/src/kentaro/articles/note/magazine/コワーキングスペース運営奮闘記/
```

ファイル名形式: テーマに合った日本語タイトル + `.md`
- 例: `HPをワードプレスにして良かった話.md`
- 例: `Tatami_Studyプログラミング教室のポスティングを頑張った話.md`

## 作成フロー

### Phase 1: 情報収集（Interactive）

1. **既存記事の確認**
   - 同フォルダ内の `.md` ファイルをGlobで検索
   - 既に作成済みのテーマを特定し、重複を防ぐ

2. **テーマの決定**

   ユーザーがテーマを指定している場合はそれを使用する。
   指定がない場合は、既存記事と重複しないテーマ候補を3-4個提案し、AskUserQuestionで選択してもらう。

   **テーマ候補の方向性**（参考）:
   - 運営上の具体的なチャレンジ（設備導入、集客、トラブル対応など）
   - 大きな意思決定や転機
   - 失敗から学んだこと
   - 新サービス・新事業の立ち上げ
   - 利用者とのエピソード

3. **深掘り質問（AskUserQuestionで1問ずつ）**

   **質問1: エピソードの背景**
   ```
   このテーマについて、背景を教えてください:

   - いつ頃の話ですか？
   - なぜそれをやろうと思ったのですか？
   - きっかけとなった出来事はありますか？

   自由に記述してください:
   ```

   **質問2: 具体的な経緯・プロセス**
   ```
   実際にどのように進めましたか？

   - 具体的にやったこと（手順、工夫、試行錯誤）
   - 関わった人（家族、利用者、業者など）
   - かかった時間やコスト（ざっくりでOK）
   - 途中で困ったこと・予想外だったこと

   自由に記述してください:
   ```

   **質問3: 結果と感想**
   ```
   結果はどうでしたか？

   - うまくいったこと / いかなかったこと
   - やってみて感じたこと・学んだこと
   - 今振り返って思うこと
   - 今後どうしていきたいか

   自由に記述してください:
   ```

   **質問4: 補足情報（オプション）**
   ```
   記事に含めたい追加情報はありますか？

   - 具体的な数値（枚数、金額、人数など）
   - 関連するURL
   - 読者に伝えたいメッセージ
   - その他エピソード

   あれば記載してください（なければスキップ）:
   ```

### Phase 2: 計画（EnterPlanMode）

1. **EnterPlanModeで記事構成を設計**

   計画ファイルに以下を記載:
   - テーマとタイトル案
   - 章立て（5-8章程度）
   - ストーリーの流れ（起承転結）

2. **ユーザー確認**
   - 構成案を提示し、修正があれば反映

### Phase 3: 執筆

1. **TaskCreateで進捗管理**

   タスク例:
   - 導入部分の作成
   - 本論の各セクション執筆
   - まとめの作成
   - 画像プレースホルダーの挿入
   - 文字数確認

2. **各セクションの執筆**

   **導入部（必須要素）**
   - テーマへの導入（問題提起や状況説明）
   - 読者の興味を引くフック
   - 1-2文で何の話かを示す

   **本論（複数セクション）**
   - 時系列またはテーマ別に展開
   - 具体的なエピソードや数値を盛り込む
   - 課題→行動→結果の流れを意識
   - 読者が追体験できるような描写

   **まとめ（必須要素）**
   - 経験から得た学びや気づき
   - 今後の展望
   - 読者へのメッセージ

3. **品質チェック**
   - 文字数: 1,500-3,000文字を目標
   - ストーリーとしての流れが自然か
   - 具体性があるか（抽象的すぎないか）
   - 画像プレースホルダーの適切な配置

### Phase 4: 保存と報告

1. **ファイル保存**
   - 正しいパスとファイル名で保存
   - 重複確認（既存ファイルの上書き防止）

2. **完了報告**
   ```
   奮闘記の記事を作成しました！

   タイトル: [タイトル]
   保存先: /home/kentaro/src/kentaro/articles/note/magazine/コワーキングスペース運営奮闘記/[ファイル名].md

   文字数: [X]文字

   次のステップ:
   - 記事を確認して必要に応じて修正
   - 画像を追加（<!-- 画像 --> コメント箇所）
   - noteに投稿
   ```

## スタイルガイド

### トーンと文体

既存記事の共通点から抽出したスタイル:

- **ストーリーテリング**: 経験をストーリーとして語る。時系列で「こうだった→こうした→こうなった」の流れ
- **正直さ**: 成功だけでなく、困難や失敗も率直に語る
- **具体性**: 抽象論ではなく、具体的な数値・エピソード・行動を中心に据える
- **等身大**: 大げさにせず、個人運営のリアルな温度感を伝える
- **前向き**: 課題があっても学びに変え、次へつなげる姿勢
- **一人称視点**: 代表の冨上（とかみ）本人の声で語る

### 構成パターン

**重要: 目次はドキュメントに含めないこと。** noteの機能で目次を追加できるため、記事本文には目次セクションを書かない。

```markdown
# タイトル

[導入: テーマの概要を1-2文で。読者の興味を引くフック]

## [セクション1: 背景・きっかけ]

[なぜこの話が始まったのか]

## [セクション2-N: 本論]

[具体的なエピソード、課題、行動、結果]

## まとめ

[学び・気づき・今後の展望]
```

### フォーマット規則

**見出し階層**
- `#` タイトル
- `##` 大セクション（目次、まとめ、各章）
- `###` サブセクション（必要な場合のみ）

**箇条書き**
- `-` を使用（`*` は不使用）
- ネストする場合は4スペースインデント

**強調**
- 重要な数値やポイントは `**太字**` で強調
- 過度な強調は避ける

**URL**
- リンクテキストを含む markdown 形式: `[テキスト](URL)`

**画像プレースホルダー**
```markdown
<!-- 画像: [説明文] -->
```

### 文字数ガイドライン

- **目標範囲**: 1,500-3,000文字
- **最小**: 1,500文字（読み応えを確保）
- **推奨**: 1,800-2,500文字（読みやすさと情報量のバランス）
- **最大**: 3,000文字（長すぎると読まれにくい）

### 表現パターン

**導入の書き出し例**
- 「コワーキングスペースのHPを、静的なサイトからWordPressへ切り替えました。」（結論先行型）
- 「プログラミング教室の開校に向けて、ポスティングに挑戦しました。」（行動先行型）
- 冒頭で何の話かを明確にし、その後に背景や詳細を展開する

**課題を述べる際**
- 具体的な状況を描写する
- 例: 「家の中だと集中できないタイプなので、カフェや図書館など行ってみましたが、混んでいたりでうまくリズムをつかむことができませんでした。」

**学びを述べる際**
- 押しつけがましくならないよう、あくまで個人の経験として語る
- 例: 「泥臭くても、地道でも、やってみなければ分からないことがある。」
- 例: 「自分でやれば効果は高い。でも、時間や労力を考えると業者に任せた方が良いこともある。」

**まとめの締め方**
- 経験を振り返った上での率直な感想で閉じる
- 奮闘記①のように「次回に続きます！」でシリーズ化する場合もある
- 教訓を一般化しすぎず、等身大のまま終わる

## 注意事項

1. **重複チェック**: 必ず既存ファイルを確認し、同じテーマの記事がないか確認する
2. **具体性**: 抽象的な話に終始せず、必ず具体的なエピソードや数値を入れる
3. **等身大**: 大げさな表現や過度な自慢・卑下を避け、リアルな温度感を保つ
5. **画像**: 実際の画像は追加できないため、コメントでプレースホルダーを示す
6. **文字数**: 文字数を確認し、目標範囲（1,500-3,000文字）内に収める
7. **質問は丁寧に**: 4段階の質問でしっかりヒアリングし、質の高い記事を作成する
8. **ナンバリング**: タイトルに「奮闘記②」等を付ける場合は、既存のナンバリングと整合させる

## エラーハンドリング

- 同じテーマの記事が既に存在する場合: ユーザーに確認し、別テーマに変更するか、切り口を変えて作成するか選択させる
- ヒアリング情報が少なすぎる場合: 追加質問を行い、具体性を確保する
- 文字数が1,500文字未満: 内容を充実させるよう提案
- 文字数が3,000文字超: 分割するか、簡潔にまとめるよう提案
